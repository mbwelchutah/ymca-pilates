You are implementing the next major product phase of my YMCA Booker app.

This app is a mobile-first gym class registration assistant.
Its highest priority is now the sniper workflow.

PRIMARY PRODUCT GOAL

The app should make it clear, before the booking window opens:

1. what class is currently targeted
2. whether the user is logged in and ready
3. whether the sniper can currently find the class
4. whether the action path is reachable
5. whether the sniper is armed and ready to book
6. what happened in the latest preflight test
7. what exact phase failed if something is blocked

This is the new center of the product.

PRIORITY ORDER

1. Now screen = Sniper Mission Control
2. Readiness / taxonomy layer
3. Preflight / Check Class Now
4. Login and session assurance
5. Sniper reliability hardening
6. Tools diagnostics alignment
7. Plan remains useful but secondary in this pass

STRICT RULES

- Keep the app runnable after every stage
- Do not redesign the whole app at once
- Do not move giant logs into Now
- Do not remove Tools diagnostics
- Keep the UI clean and iPhone-oriented
- Keep detailed evidence in Tools
- Show only concise readiness and mission state on Now
- Stop after each stage and summarize exactly what changed
- Do not jump ahead

IMPORTANT PRODUCT SHIFT

The Now screen is no longer just a passive countdown screen.
It should become a sniper mission-control screen.

When the user opens Now, they should be able to tell:
- Am I logged in?
- Can the sniper find the class?
- Can it reach the booking action?
- Is the sniper waiting, ready, armed, booking, blocked, or recovering?
- What happened the last time I tested it?

STAGE 2 — CONNECT TAXONOMY TO EXISTING SNIPER FLOW
================================

Goal:
Make the real sniper path emit readiness and phase-based failures.

Tasks:
1. At auth/session checks:
   emit SESSION_READY / SESSION_REQUIRED / SESSION_EXPIRED
2. At navigation:
   emit navigation-phase failures if relevant
3. At class discovery:
   emit DISCOVERY_READY / DISCOVERY_FAILED / DISCOVERY_NOT_TESTED
4. At verify:
   emit readiness/failure impacts as appropriate
5. At modal/action detection:
   emit ACTION_READY / ACTION_BLOCKED and relevant failures
6. Derive sniper state from real readiness bundle:
   - if session + discovery + action are ready -> SNIPER_READY or SNIPER_ARMED
   - if auth fails -> SNIPER_BLOCKED_AUTH
   - if discovery fails -> SNIPER_BLOCKED_DISCOVERY
   - if action fails -> SNIPER_BLOCKED_ACTION
7. Ensure this uses real checks, not fake UI-only states

Do not redesign screens yet.

After Stage 2:
Stop and summarize:
- where readiness is now emitted
- which sniper code points are instrumented
- how sniper state is derived

STAGE 4 — PREFLIGHT / CHECK CLASS NOW
================================

Goal:
Allow the user to safely test the sniper path before the booking window opens.

Tasks:
1. Add a clearly labeled action on Now:
   - Check Class Now
   - or Run Preflight
2. This action should:
   - validate login/session
   - navigate to the schedule
   - apply filters if needed
   - find the target class
   - verify title/instructor/time/date
   - optionally open modal safely
   - detect Reserve / Waitlist / Closed state
3. The preflight must stop before actual booking unless explicitly allowed
4. The preflight must produce a structured result using the taxonomy
5. Show a concise result on Now:
   - Preflight passed
   - Login required
   - Class not found
   - Action blocked
   - Modal issue
6. Store detailed evidence in Tools

Important:
This is a readiness check, not a real booking attempt.
