==================================================
CONTEXT / PROBLEMS TO FIX
==================================================

Known issues from the post-refactor audit:

A. Warmup classifier gate ignores freshness
- tick.js suppresses a warmup run when classifier says state === 'full'
- it does not also require freshness to be fresh
- this means an aging/stale "full" result can suppress the only useful warmup run

B. Write-order lag
- refreshReadiness() runs before refreshConfirmedReadyState()
- readiness-state.json reads confirmed-ready-state.json from disk
- therefore readiness can be one run behind confirmed-ready truth

C. Ping fast-path does not refresh confirmed-ready state
- when auto-preflight succeeds via ping + booking confirmed + fresh cache
- confirmed-ready-state.json may not be rewritten
- freshness silently ages even though the system just validated truth

We want to fix these without introducing new architecture complexity.

STAGE 1 — FIX WARMUP CLASSIFIER GATE TO RESPECT FRESHNESS
==================================================

Goal:
Do not let aging/stale class truth suppress warmup runs.

Tasks:

1. Find the warmup classifier gate in tick.js where the run is skipped if class state is full
2. Change the logic so a warmup skip happens ONLY when:
   - classifier state === 'full'
   AND
   - classifier freshness === 'fresh'

3. If classifier state is full but freshness is:
   - aging
   - stale
   - unknown
then DO NOT suppress the warmup run automatically

4. In those non-fresh cases:
   - allow the warmup run to continue
   OR
   - downgrade the classifier result to effectively “unknown” for gate purposes

5. Keep fresh “full” cache useful, but make non-fresh “full” non-authoritative for warmup suppression

Success criteria:
- aging/stale "full" can no longer block a useful warmup run
- only fresh full-state truth is authoritative enough to suppress warmup
