You are implementing a staged UX refinement for the Run Check experience on the Now screen of my YMCA Booker app.

PRIMARY GOAL

Make Run Check feel:
- continuous
- calm
- trustworthy
- informative

Right now the logic works, but the experience feels too abrupt:
- it jumps from “Checking…” to a red failure-like card
- the wording feels harsher than the situation
- the user cannot see progress unfolding step by step

This should be improved without redesigning the whole app.

IMPORTANT PRODUCT DIRECTION

Most Run Check outcomes are not true failures.
Very often the app is working correctly, but registration is simply not open yet.

The UI should communicate:
- progress while checking
- the difference between “working but not open” and “actually broken”
- confidence and reassurance

STRICT RULES

- Keep the app runnable after every stage
- Do not redesign the whole Now screen
- Do not remove existing logic
- Do not remove detailed diagnostics from Tools
- Do not add raw logs to Now
- Stop after each stage and summarize exactly what changed
- Do not jump ahead

================================
STAGE 1 — STEP-BY-STEP CHECK PROGRESS
================================

Goal:
Replace the single generic “Checking…” state with a sequence of visible check steps.

Tasks:
1. During Run Check, show meaningful progress text instead of just “Checking…”
2. Use a step sequence such as:
   - Checking session…
   - Checking schedule…
   - Finding class…
   - Opening booking…
   - Checking availability…
3. Update the Now screen as each step runs
4. Keep the UI calm and simple
5. Do not add debug detail or raw technical traces

Important:
The user should feel that the system is actively working through the check, not just hanging.

After Stage 1:
Stop and summarize:
- what check steps are shown
- where they appear
- how they update during a run

================================
STAGE 2 — PERSIST STEP RESULTS VISUALLY
================================

Goal:
Show the outcome of each major step after the check completes.

Tasks:
1. When Run Check completes, keep simple user-facing results for major steps such as:
   - Session: Ready
   - Class: Found
   - Modal: Reachable
   - Action: Not open yet / Ready / Waitlist only
2. Use concise human-readable wording
3. Keep this in the current compact readiness/details area if possible
4. Do not turn the screen into a debug dashboard

Important:
The user should be able to see that the system succeeded at several steps even if the final action is not yet available.

After Stage 2:
Stop and summarize:
- what step results persist after the run
- where they appear
- how this differs from the old binary result

================================
STAGE 3 — SOFTEN HARSH WORDING
================================

Goal:
Replace unnecessarily harsh or failure-like wording with calmer, more accurate language.

Tasks:
1. Replace harsh phrases such as:
   - “Action blocked”
   - “Booking action unavailable”
   - “No booking button visible”
2. Use calmer, more accurate alternatives such as:
   - “Not open yet”
   - “Booking not available yet”
   - “Registration hasn’t opened yet”
3. Keep real error language only for real failures:
   - login failure
   - session expired
   - class not found
   - unexpected modal failure
4. Make sure the wording matches the actual state

Important:
If the system is working and the class simply is not open yet, the message should feel expected, not alarming.

After Stage 3:
Stop and summarize:
- what phrases were changed
- what the new user-facing wording is
- how true failures are still distinguished
